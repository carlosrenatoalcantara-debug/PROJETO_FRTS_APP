## END-TO-END OPERATIONAL FLOW

### Complete Workflow (STEP 1-2 Integrated)

```
┌─────────────────────────────────────────────────────────────────┐
│ BILL INTAKE + HOMOLOGATION OPERATIONAL FLOW                     │
└─────────────────────────────────────────────────────────────────┘

1. BILL UPLOAD
   ↓
   User uploads PDF fatura.pdf
   ↓
   /api/bills/upload (multipart form-data)
   
2. VALIDATION
   ↓
   billIntakeService.validateBillFile()
   - Check file existence
   - Verify size ≤ 10MB
   - Confirm PDF format
   
3. TEXT EXTRACTION
   ↓
   neoenergiaParser.extractTextFromPDF()
   - Use PDF text layer (primary)
   - Fallback: OCR handling
   - Normalize whitespace
   
4. DETERMINISTIC PARSING
   ↓
   neoenergiaParser.parseNeoenergiaBill()
   - parseClientData() → regex extraction
   - parseUnitData() → UC technical fields
   - parseConsumptionHistory() → 12-month history
   - Numeric normalization (Brazilian locale)
   
5. CONFIDENCE SCORING
   ↓
   calculateExtractionConfidence()
   - Score: 85-100% (high confidence)
   - Score: 70-85% (review before creation)
   - Score: <70% (manual correction needed)
   
6. DTO VALIDATION & FREEZING
   ↓
   validateBillExtractedData()
   - Validate all required fields
   - Normalize data types
   - Freeze via deepFreezeSafe()
   
7. RESPONSE TO FRONTEND
   ↓
   {
     "sucesso": true,
     "arquivo": "fatura_neoenergia.pdf",
     "distribuidora": "neoenergia_cosern",
     "extraido": {
       "cliente": {...},
       "unidade": {...},
       "consumo": {...},
       "historico_consumo": [...]
     },
     "confianca_extracao": 95,
     "timestamp_processamento": "2026-05-23T14:30:00Z"
   }

8. AUTOMATIC CLIENT CREATION (STEP 2 - Database Integration Pending)
   ↓
   Frontend receives extracted data
   User reviews/confirms extracted values
   
   - Check for duplicate client (cpf_cnpj_parcial)
   - Create new client or link to existing
   - Store client contact info (nome, endereco, etc.)
   - Create Unidade Consumidora record
   - Link to client
   
9. AUTOMATIC PROJECT INITIALIZATION (STEP 2 - Database Integration Pending)
   ↓
   From consumption profile:
   - Create FV project container
   - Pre-populate from extraction:
     * consumo_medio → sizing baseline
     * consumo_12_meses → seasonality analysis
     * classe_tarifaria → tariff group
     * tensao → voltage for equipment selection
   - Ready for FV engine
   
10. RESPONSE
    ↓
    {
      "cliente_id": "abc123",
      "uc_id": "def456",
      "projeto_id": "ghi789",
      "status": "PRONTO_PARA_DIMENSIONAMENTO",
      "proximo_passo": "Revisar dados de consumo e iniciar FV sizing"
    }
```

## IMMUTABILITY GUARANTEES

All extracted data is immutable at runtime:

```javascript
// Exemplo do que NÃO é possível fazer
const resultado = await billIntakeService.processBill(...)

// ❌ TypeError - Object is frozen
resultado.extraido.cliente.nome_cliente = "NOVO NOME"

// ❌ TypeError - Array is frozen
resultado.extraido.consumo.consumo_12_meses[0] = 999

// ❌ TypeError - Nested object is frozen
resultado.extraido.consumo.sazonalidade.desvio_padrao = 0
```

**Guarantee**: Engineering truth from bill extraction remains immutable.
